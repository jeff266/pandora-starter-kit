## T010 — safeExecute Empty Result Warning (NEW)

Context: Three consecutive sprints have had silent compute failures where 
safeExecute caught an error or a tool returned an empty result — no throw, 
no log, just missing data downstream. Examples:
  - persistAccuracyScores reading context.stepOutputs → returned nothing silently
  - enrichForecastWithAccuracy querying wrong columns → byRep: {} silently
  - loadVelocityBenchmarks using wrong table/column names → empty silently

Find safeExecute in server/skills/tool-definitions.ts (or wherever it lives).

Current behavior (approximate):
  async function safeExecute(toolName, fn, params) {
    try {
      return await fn();
    } catch (err) {
      logger.error(`[Tool Error] ${toolName}: ${err.message}`);
      return null; // or {} or []
    }
  }

Add two enhancements:

1. Empty result warning — after a successful execution (no throw), 
   check if the result is meaningfully empty:

  const result = await fn();
  
  const isEmpty = 
    result === null || 
    result === undefined ||
    (Array.isArray(result) && result.length === 0) ||
    (typeof result === 'object' && !Array.isArray(result) && 
     Object.keys(result).length === 0);
  
  if (isEmpty) {
    logger.warn(
      `[ToolWarning] ${toolName} returned empty result — ` +
      `possible schema mismatch or no data for this workspace`
    );
  }
  
  return result;

2. Execution time logging — log how long each tool took:

  const start = Date.now();
  const result = await fn();
  const ms = Date.now() - start;
  
  if (ms > 5000) {
    logger.warn(`[ToolSlow] ${toolName} took ${ms}ms — consider query optimization`);
  }

These are warn-level logs only — never throw, never change return value. 
The existing behavior is preserved; we're adding observability only.

After adding: restart server and run deal-risk-review on Frontera. 
Confirm the startup log still shows 30 skills registered. Confirm no 
new errors from the safeExecute change.
