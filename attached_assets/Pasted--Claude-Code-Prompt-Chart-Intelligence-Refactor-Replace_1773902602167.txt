Claude Code Prompt: Chart Intelligence Refactor —
Replace DeepSeek with Two-Step Haiku + Rules

## Context

The current chart intelligence system uses DeepSeek
to classify which charts to generate. DeepSeek
pattern-matches to common chart types from training
data — producing pie charts showing "23% of pipeline
converted" when the VP needs "which deals can close
in 12 days."

The problem is architectural: DeepSeek is good at
structured classification (yes/no, category A/B/C)
but lacks the business reasoning to answer "what
does a VP RevOps need to see to make a decision?"

This prompt refactors to a two-step architecture:

Step 1 — Claude Haiku reasons about the decision
  "What question must this chart answer for
   the VP to take action?"
  Output: a structured chart question spec

Step 2 — Rules resolve the spec to a chart
  Given the question type, select data, type,
  colors, orientation deterministically.
  No model variance. Fully testable.

DeepSeek is removed from chart intelligence entirely.

Before starting, read ALL of these:

1. server/orchestrator/chart-intelligence.ts —
   full file. Understand generateChartSpecs(),
   classifyChartForNode(), buildDeepSeekPrompt(),
   resolveDataPoints(), applyOrientationRule().
   You are replacing the DeepSeek call in
   classifyChartForNode() with a Haiku call,
   and replacing resolveDataPoints() with a
   deterministic resolver driven by question type.

2. server/orchestrator/types.ts — ChartNodeSpec,
   ChartDataPoint, ReasoningNode, AtRiskDeal,
   StaleDeal, SkillSummary. You are adding
   ChartQuestion interface.

3. server/llm/router.ts — find how Claude Haiku
   is called. Look for capability: 'reason' or
   a model parameter that routes to Haiku.
   If Haiku isn't explicitly available, use
   Claude with maxTokens: 400 and a note that
   this should route to Haiku when available.

4. server/orchestrator/chart-renderer.ts —
   the SEMANTIC_COLORS map. You are extending it.

5. The current failing chart: "Only 23% of pipeline
   converted to revenue despite massive opportunity"
   — this is a 2-item key_metrics chart (closed_won
   vs remaining_pipeline) that should instead be
   a deal-timing chart showing days until close
   vs days since last activity per deal.

Do not write code until all five are read.

## New Architecture

```
ReasoningNode + SkillSummary
        ↓
Step 1: Haiku reasons about VP decision
  Input: node question, node answer,
         available data summary
  Output: ChartQuestion {
    vp_decision: string,    // what must VP decide?
    chart_question: string, // what does chart answer?
    question_type: enum,    // drives data selection
    preferred_data: string, // which data source
    preferred_type: string, // bar/horizontal/donut/line
  }
        ↓
Step 2: Rules resolve ChartQuestion to ChartNodeSpec
  - question_type → data source selection
  - data source → actual ChartDataPoint[]
  - label length → orientation override
  - data semantics → color encoding
  - item count → legend visibility
  - line chart guard → block if non-time-series
        ↓
ChartNodeSpec (deterministic, rules-based)
        ↓
QuickChart renders PNG
```

export type ChartQuestionType =
  | 'deal_triage'
  // Which specific deals to work vs abandon?
  // → use at_risk_deals[], horizontal bar,
  //   sorted by urgency (days until close)

  | 'deal_timing'
  // When are deals closing relative to activity?
  // → use at_risk_deals[], horizontal bar,
  //   showing days_stale or days_in_stage

  | 'rep_comparison'
  // How do reps compare on a metric?
  // → use rep_performance data, bar chart,
  //   teal=primary rep, amber=secondary

  | 'pipeline_composition'
  // What portion of pipeline is in each state?
  // → use key_metrics, donut chart
  // ONLY when composition IS the argument
  // (not as default fallback)

  | 'coverage_gap'
  // How far are we from a target?
  // → comparative bar: actual vs target
  // two bars, teal=actual, gray=target

  | 'trend'
  // How has a metric changed over time?
  // → line chart, requires ≥3 time points
  // BLOCKED if data has < 3 points

  | 'metric_comparison'
  // Comparing 3+ discrete values of same type?
  // → vertical bar, uniform color

  | 'not_chartable'
  // Qualitative argument, no numeric data,
  // or data_gap = true
  // → no chart generated

export interface ChartQuestion {
  vp_decision: string;
  // What must the VP decide after reading this?
  // e.g. "Which deals to close vs abandon today"

  chart_question: string;
  // What specific question does the chart answer?
  // e.g. "Which deals close within 14 days
  //        and have had recent activity?"

  question_type: ChartQuestionType;

  preferred_data: 'at_risk_deals' | 'stale_deals'
    | 'rep_performance' | 'key_metrics' | 'none';

  reasoning: string;
  // Why this chart type for this argument
  // (for logging and debugging)
}

/**
 * Step 1: Haiku reasons about what question
 * the chart must answer for the VP to act.
 *
 * This is the ONLY LLM call in chart intelligence.
 * It outputs a ChartQuestion, not a ChartNodeSpec.
 * Rules in step 2 resolve the spec deterministically.
 */
async function reasonAboutChart(
  node: ReasoningNode,
  availableData: AvailableDataSummary,
  workspaceId: string
): Promise<ChartQuestion> {

  const systemPrompt = `You are advising a VP of
Revenue Operations who must make a decision today.

Your job: given a reasoning node from a sales
intelligence report, decide what question a chart
should answer — if a chart is warranted at all.

A chart is warranted when it makes a specific
decision OBVIOUS in 3 seconds. It is NOT warranted
when it merely describes a state the VP already
knows from reading the text.

The worst charts:
- Show portfolio composition as a default
  (pie charts showing % closed vs % open)
- Describe history when the VP needs to act now
- Answer a question nobody asked

The best charts for late-quarter deal sections:
- Show which specific deals can close vs cannot
- Show how long deals have been silent
- Show which rep owns the risk

Output ONLY valid JSON. No explanation.`;

  const userMessage = `REASONING NODE:
Layer: ${node.layer}
Question: "${node.question}"
Answer: "${node.answer.slice(0, 400)}"

AVAILABLE DATA:
${availableData.atRiskDeals.length > 0
  ? `At-risk deals (${availableData.atRiskDeals.length}):
${availableData.atRiskDeals.map(d =>
    `  ${d.name} $${Math.round(d.amount/1000)}K ` +
    `risk:${d.risk_score} ${d.days_in_stage}d in stage`
  ).join('\n')}`
  : 'No at-risk deals'}

${availableData.staleDeals.length > 0
  ? `Stale deals (${availableData.staleDeals.length}):
${availableData.staleDeals.map(d =>
    `  ${d.name} $${Math.round(d.amount/1000)}K ` +
    `${d.days_stale}d dark`
  ).join('\n')}`
  : 'No stale deal data'}

${availableData.repData.length > 0
  ? `Rep data:\n${availableData.repData.map(r =>
    `  ${r.name} $${Math.round(r.pipeline/1000)}K`
  ).join('\n')}`
  : ''}

${availableData.keyMetrics.length > 0
  ? `Key metrics:\n${availableData.keyMetrics
    .slice(0, 5).map(m => `  ${m.key}: ${m.value}`)
    .join('\n')}`
  : ''}

Respond ONLY with JSON:
{
  "vp_decision": "one sentence: what must VP decide?",
  "chart_question": "one sentence: what does chart answer?",
  "question_type": "deal_triage|deal_timing|rep_comparison|pipeline_composition|coverage_gap|trend|metric_comparison|not_chartable",
  "preferred_data": "at_risk_deals|stale_deals|rep_performance|key_metrics|none",
  "reasoning": "one sentence: why this type"
}`;

  // Call Claude (Haiku when available, Sonnet as fallback)
  // This is a classification call — max 300 tokens output
  const response = await callLLM(
    workspaceId, 'reason', {
    systemPrompt,
    messages: [{ role: 'user', content: userMessage }],
    maxTokens: 300,
    temperature: 0.1,
    model: 'claude-haiku-4-5-20251001',
    // Explicit Haiku — cheap, fast, sufficient
    // for structured JSON classification
    _tracking: {
      workspaceId,
      skillId: 'chart-intelligence',
      phase: 'reason',
      stepName: `chart-question-${node.layer}`,
    },
  });

  const raw = response.content
    .map((b: any) => b.text || '')
    .join('')
    .replace(/```json\s*/g, '')
    .replace(/```\s*/g, '')
    .trim();

  const parsed = JSON.parse(raw);

  console.log(
    `[ChartIntelligence] ${node.layer}: ` +
    `${parsed.question_type} — ${parsed.reasoning}`
  );

  return parsed as ChartQuestion;
}

/**
 * Step 2: Rules resolve ChartQuestion to actual
 * data points. Fully deterministic — no model.
 *
 * question_type drives data selection.
 * Data shape drives orientation and colors.
 * Guards block nonsensical chart types.
 */
function resolveChartFromQuestion(
  question: ChartQuestion,
  skillSummaries: SkillSummary[]
): Omit<ChartNodeSpec, 'should_chart'> | null {

  const riskSkill = skillSummaries.find(
    s => s.skill_id === 'deal-risk-review'
  );
  const hygieneSkill = skillSummaries.find(
    s => s.skill_id === 'pipeline-hygiene'
  );
  const coverageSkill = skillSummaries.find(
    s => s.skill_id === 'pipeline-coverage'
  );

  switch (question.question_type) {

    case 'deal_triage':
    case 'deal_timing': {
      // Use at_risk_deals sorted by risk score
      // Show deal name + amount
      // Color by risk: dead=red, at_risk=amber, live=teal
      const deals = riskSkill?.at_risk_deals || [];
      if (deals.length < 2) return null;

      const dataPoints: ChartDataPoint[] = deals
        .sort((a, b) => b.risk_score - a.risk_score)
        .slice(0, 6)
        .map(d => ({
          label: d.name,
          value: Math.round(d.amount / 1000),
          semantic_tag: d.risk_score > 80 ? 'dead'
            : d.risk_score > 60 ? 'at_risk'
            : 'healthy',
        }));

      return {
        chart_type: 'horizontal_bar',
        // Always horizontal for named deals
        title: question.chart_question,
        data_points: dataPoints,
        color_scheme: 'semantic',
        show_legend: false,
        reasoning: question.reasoning,
      };
    }

    case 'rep_comparison': {
      // Rep pipeline comparison
      // Bar chart, teal=primary, amber=secondary
      const repData = buildRepData(coverageSkill);
      if (repData.length < 2) return null;

      return {
        chart_type: 'bar',
        title: question.chart_question,
        data_points: repData.map((r, i) => ({
          label: r.name,
          value: Math.round(r.pipeline / 1000),
          semantic_tag: i === 0 ? 'actual' : 'target',
        })),
        color_scheme: 'uniform',
        show_legend: false,
        reasoning: question.reasoning,
      };
    }

    case 'coverage_gap': {
      // Actual vs target — two bars
      const m = coverageSkill?.key_metrics || {};
      const actual = Number(
        m['pipeline-coverage.total_pipeline']
        || m['total_pipeline'] || 0
      );
      const target = Number(
        m['pipeline-coverage.quota']
        || m['quota'] || 0
      );
      if (actual === 0 && target === 0) return null;

      return {
        chart_type: 'bar',
        title: question.chart_question,
        data_points: [
          { label: 'Open Pipeline',
            value: Math.round(actual / 1000),
            semantic_tag: 'actual' },
          { label: 'Coverage Target',
            value: Math.round(target / 1000),
            semantic_tag: 'target' },
        ],
        color_scheme: 'comparative',
        show_legend: false,
        reasoning: question.reasoning,
      };
    }

    case 'pipeline_composition': {
      // Donut — ONLY when composition is the argument
      // Not as a default fallback
      const m = coverageSkill?.key_metrics || {};
      const closed = Number(
        m['forecast-rollup.closed_won']
        || m['closed_won'] || 0
      );
      const open = Number(
        m['pipeline-coverage.total_pipeline']
        || m['total_pipeline'] || 0
      );
      if (closed === 0 && open === 0) return null;

      return {
        chart_type: 'donut',
        title: question.chart_question,
        data_points: [
          { label: 'Closed Won',
            value: Math.round(closed / 1000),
            semantic_tag: 'positive' },
          { label: 'Open Pipeline',
            value: Math.round(open / 1000),
            semantic_tag: 'target' },
        ],
        color_scheme: 'semantic',
        show_legend: true,
        reasoning: question.reasoning,
      };
    }

    case 'trend': {
      // Line chart — BLOCKED if < 3 data points
      // Most ad-hoc questions won't have time series
      // This type is reserved for prior_context data
      // which has week-over-week history
      console.log(
        '[ChartIntelligence] Trend chart requested ' +
        'but no time series data — skipping'
      );
      return null;
    }

    case 'not_chartable':
    default:
      return null;
  }
}

function buildRepData(
  coverageSkill: SkillSummary | undefined
): { name: string; pipeline: number }[] {
  if (!coverageSkill?.key_metrics) return [];

  const reps: { name: string; pipeline: number }[] = [];
  const m = coverageSkill.key_metrics;

  for (const [key, val] of Object.entries(m)) {
    if ((key.includes('rep') || key.includes('owner'))
        && key.includes('pipeline')
        && !key.includes('total')) {
      const name = key
        .replace(/_pipeline|_open/g, '')
        .replace(/_/g, ' ')
        .trim();
      const value = Number(val);
      if (value > 0 && name.length > 0) {
        reps.push({ name, pipeline: value });
      }
    }
  }

  return reps.sort((a, b) => b.pipeline - a.pipeline)
    .slice(0, 4);
}

## Task 5: Wire it together

Replace the body of classifyChartForNode() to

```typescript
async function classifyChartForNode(
  node: ReasoningNode,
  skillSummaries: SkillSummary[],
  workspaceId: string
): Promise<ChartNodeSpec | null> {

  const availableData = buildAvailableDataBlock(
    skillSummaries
  );

  if (!availableData.hasNumericData) return null;

  // Step 1: Haiku reasons about VP decision
  const question = await reasonAboutChart(
    node, availableData, workspaceId
  );

  if (question.question_type === 'not_chartable') {
    console.log(
      `[ChartIntelligence] ${node.layer}: ` +
      `not chartable — ${question.reasoning}`
    );
    return null;
  }

  // Step 2: Rules resolve to chart spec
  const spec = resolveChartFromQuestion(
    question, skillSummaries
  );

  if (!spec || spec.data_points.length < 2) {
    return null;
  }

  // Apply orientation guard (label length rule)
  const hasLongLabel = spec.data_points.some(
    d => d.label.length > 12
  );
  if (hasLongLabel &&
      spec.chart_type !== 'donut' &&
      spec.chart_type !== 'line') {
    spec.chart_type = 'horizontal_bar';
  }

  return {
    should_chart: true,
    ...spec,
    title: question.chart_question,
  };
}
```
