## T001 Validation — RFM Signal in Deal Risk Review

First, confirm the data source:
  SELECT column_name FROM information_schema.columns 
  WHERE table_name = 'deals' 
  AND column_name LIKE '%rfm%'

  AND separately:
  SELECT column_name FROM information_schema.columns 
  WHERE table_name = 'lead_scores'
  AND column_name LIKE '%rfm%'

If rfm fields exist on the deals table: confirm they are populated for 
Frontera workspace (SELECT COUNT(*) FROM deals WHERE workspace_id = [id] 
AND rfm_grade IS NOT NULL). If count is 0, the enrich step will silently 
return nulls for every deal and the RFM wiring provides no value.

If rfm fields only exist on lead_scores: the enrichDealsWithRfm function 
is querying the wrong table. Fix it to JOIN deals to lead_scores on 
entity_id = deal_id WHERE entity_type = 'deal'.

After confirming the data source is correct and populated, run 
deal-risk-review on Frontera. Check the skill_run step outputs:
  - enrich-with-rfm-scores step should return non-null rfm_grade for 
    at least some deals
  - classify-deal-risk DeepSeek input should include rfm_grade per deal
  - Claude synthesis should reference at least one RFM grade explicitly

If rfm_grade is null for all deals: the data isn't there yet — note it 
but don't block. The wiring is correct; the scoring engine needs to run 
first to populate lead_scores.

## T002 Validation — Forecast Model Calibration Gate

Run checkForecastCalibration against both Imubit and Frontera:
  POST /api/workspaces/[imubit_id]/skills/forecast-model/run
  POST /api/workspaces/[frontera_id]/skills/forecast-model/run

For each, check the skill_run step output for calibration_tier value.

Expected:
  Imubit (Salesforce, mature data): should hit 'full' or 'partial'
  Frontera (HubSpot, mid-market): confirm which tier

If either workspace hits 'insufficient' unexpectedly: the thresholds 
(30+ closed deals, 10+ won) may be too aggressive for mid-market. 

Check actual counts:
  SELECT 
    COUNT(*) as total_closed,
    COUNT(*) FILTER (WHERE is_closed_won = true) as closed_won
  FROM deals 
  WHERE workspace_id = $1
    AND closed_at > NOW() - INTERVAL '180 days'

If a workspace with meaningful data hits 'insufficient', lower the 
full tier threshold to match the original spec: 10+ closed-won deals, 
6+ months. Update checkForecastCalibration accordingly.

## T003 Validation — Pipeline Gen Forecast History Gate

Same pattern. Run against Frontera and Imubit.

Check actual pipeline creation history:
  SELECT COUNT(DISTINCT DATE_TRUNC('month', created_at)) as months_with_deals,
         COUNT(*) as total_deals
  FROM deals
  WHERE workspace_id = $1
    AND created_at > NOW() - INTERVAL '90 days'

If a workspace with 6+ months of data hits 'insufficient' (threshold: 
40+ deals / 90 days), the window is too short. Original spec used 
monthly cohorts over 6 months — reconsider whether 90 days is the 
right lookback. If fewer than 3 months of data are found, 
'insufficient' is correct. If 6+ months exist but the 90-day window 
misses them, extend to 180 days.
