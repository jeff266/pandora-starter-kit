The Sales Roster (rep hierarchy)
The Methodology config (qualification framework)
The Segments table (business segments)
The Pipeline Config (value fields per pipeline)
So when a user asks "what's our coverage ratio?" Pandora computes pipeline / some default target — not pipeline / the actual Q2 target that's sitting in the Targets table. It's running analysis without reading the configuration that defines what the analysis should mean.

That's the core gap. The tools exist. The data exists. The connection between "user intent" and "workspace-specific definition" is missing.

The agentic learning problem — what happens when specifics aren't in structured settings?

This is where it gets interesting. Three categories of context that Pandora encounters in conversations:

Category 1: Correctable structured settings User says "actually our win rate benchmark is 35%, not 30%." This should update metric_definitions. Already partially solved by calibration. The pattern: detect the correction, write to the canonical table, confirm back to the user.

Category 2: Business context that has no structured home User says "our ABA therapy deals take 90 days minimum because of insurance authorization." There's no table for that. But it's critical context — it means Pandora shouldn't flag a 60-day deal as stale, and it means win rate benchmarks need to account for cycle time. This is the hard problem.

Category 3: Implicit preferences revealed through use User always asks about Core Sales Pipeline but never mentions Fellowship Pipeline. User always filters by Nate's deals. User always asks about Q2 close dates. These aren't corrections or statements — they're revealed preferences that Pandora can infer from usage patterns.

The mechanism for learning and storing unstructured context:

The right architecture is a three-tier memory system:

Tier 1 — Structured workspace configuration (what exists today) workspace_config, business_dimensions, metric_definitions, Targets table, Pipeline Config, Sales Roster. Fast lookup, canonical, affects all computations. Modified through calibration + direct settings edits.

Tier 2 — Workspace knowledge base (needs building) A workspace_knowledge table. Free-form key-value pairs with a source, confidence score, and last-referenced timestamp. Every time Pandora learns something in conversation that doesn't fit Tier 1, it writes here.

sql
workspace_knowledge (
  workspace_id,
  key,           -- e.g. 'deal_cycle_constraint.insurance_auth'
  value,         -- e.g. 'ABA therapy deals require 90d minimum 
                 --       due to insurance authorization process'

  last_used_at,
  used_count

```

When a user says "our ABA therapy deals take 90 days minimum because of insurance authorization" — Pandora extracts the claim, writes it to `workspace_knowledge`, and uses it in every subsequent analysis. Stale deal alerts for ABA deals now use 90 days, not 14. Coverage analysis notes the cycle time constraint.

**Tier 3 — Conversation patterns** (partially exists via training data capture)
Usage patterns extracted from conversation history. Which pipelines get queried most. Which reps get focused on. Which time periods matter. These inform defaults but don't override structured config.
```

No. And this is one of the most important design decisions in the whole system.

The problem with blind write-back:

If a user says "our win rate is 30%" and Pandora immediately overwrites metric_definitions, you've replaced a computed value with an asserted one. Pandora actually computed 30% from 122 won / 349 closed. That's a fact. The user saying "our win rate is 30%" might mean:

Confirming what Pandora already computed — safe to write
Stating a target or aspiration, not an actuality — dangerous to write
Repeating something they heard in a meeting that was wrong — dangerous to write
Referring to a different time period than Pandora computed — wrong to overwrite without context
Referring to a different pipeline than Pandora computed — wrong to overwrite globally
The three-tier confidence model:

TIER 1 — COMPUTED (highest trust)
  Source: Pandora computed from CRM data
  Example: win rate = 30% (122/349, last 12 months)
  Write behavior: Always authoritative. Never overwritten 
  by assertion alone.

TIER 2 — CONFIRMED (high trust)  
  Source: User reviewed Pandora's computed value 
  and agreed with it
  Example: "Yes, 30% sounds right"
  Write behavior: Locks the value as confirmed. 
  Pandora stops recomputing and uses this number.

TIER 3 — ASSERTED (lower trust)
  Source: User stated a value without Pandora 
  computing it first
  Example: "Our win rate is 30%"
  Write behavior: Stored in workspace_knowledge 
  with confidence=0.6, NOT written to 
  metric_definitions. Surfaced for verification.
What Pandora should do when a user asserts a metric value:

If Pandora has a computed value for that metric, show the comparison:

"You mentioned your win rate is 30%. That matches what I computed — 122 wins out of 349 closed deals over the last 12 months. Want me to lock this in as your official win rate benchmark?"

If user says yes → write to metric_definitions as CONFIRMED, confidence=1.0.

If Pandora's computed value differs — say it computed 24%:

"You mentioned 30%, but from your CRM data I'm computing 24% over the last 12 months (89 won / 371 closed). A few things could explain the gap: different time period, different pipeline scope, or deals that closed after your count. Which should I use?"

Now the user has to make an active choice with the discrepancy surfaced. They can't accidentally overwrite a computed value with a casual assertion.

If Pandora has no computed value at all (metric doesn't exist in CRM data):

"I can't verify that from your CRM — I don't have enough closed-lost data to compute it myself. I'll store 30% as your stated win rate and use it for coverage calculations. You can update it anytime."

This is the only case where an assertion writes directly to metric_definitions — when Pandora has no basis to contradict it.

The write-back rules:

User asserts value + Pandora has computed value
  → Show comparison
  → Ask for explicit confirmation
  → Write only on confirmation

User asserts value + Pandora computed value matches
  → Still ask for confirmation (closes the loop)
  → Write on confirmation

User asserts value + Pandora has no computed value
  → Write with source='asserted', confidence=0.6
  → Flag as unverified in Data Dictionary
  → Recompute when data becomes available

User corrects a CONFIRMED value
  → "You previously confirmed 30%. You're now saying X. 
     Should I update the benchmark?"
  → Write only on explicit yes
  → Log the change with reason

Pandora computes a new value that differs from 
a CONFIRMED value
  → Surface the discrepancy in next brief
  → Do NOT auto-update the confirmed value
  → Let the user decide

Logged. This principle has a name in epistemology — it's the difference between testimony and evidence. Pandora should trust evidence over testimony, and require explicit confirmation to elevate testimony to the level of evidence.

The practical implication is also what makes Pandora trustworthy in a board meeting. If a CRO asks "what's our win rate?" and Pandora says 30% — they need to know whether that's:

Computed — Pandora counted it from deal history. Cite-able.
Confirmed — Pandora computed it, you agreed. Cite-able.
Asserted — Someone told Pandora. Not cite-able without verification.
The Data Dictionary source badge already captures this distinction — SYSTEM vs USER. Extending it to COMPUTED / CONFIRMED / ASSERTED gives every metric a full audit trail. Anyone can see not just what the number is, but how Pandora knows it.

That's the difference between a reporting tool that shows numbers and an intelligence platform that you'd stake a forecast on.
