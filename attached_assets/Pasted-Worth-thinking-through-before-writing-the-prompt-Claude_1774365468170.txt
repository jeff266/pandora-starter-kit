Worth thinking through before writing the prompt. Claude Code can absolutely do React/TypeScript — it's not limited to backend. The distinction is really about complexity and coordination. The chart builder redesign is a large multi-file React refactor that touches:

- `ReportViewer.tsx` — inline expansion replacing the drawer
- `ChartBuilder.tsx` — the entire builder component rebuilt with the three-zone model
- Possibly new component files for the field picker, chart type grid, config panel sections
- CSS/styles

Claude Code handles this well when the spec is tight. The risk with complex UI builds in Claude Code is visual QA — it can't see what it's rendering. The mockup we built earlier helps significantly because Claude Code can reference the exact layout, colors, and interaction patterns without guessing.

  { type: 'scatter',      label: 'Scatter',
    requires: 'second_y',
    tooltip: 'Add a second Y axis above' },
  { type: 'waterfall',    label: 'Waterfall',
    requires: 'skill_data',
    tooltip: 'Use Skills tab — requires computed data' },
  { type: 'bullet',       label: 'Bullet',
    requires: 'skill_data',
    tooltip: 'Use Skills tab — requires target data' },
]

function isChartTypeEnabled(
  type: ChartTypeConfig,
  hasSeriesField: boolean,
  hasSecondY: boolean
): boolean {
  if (type.requires === 'none') return true
  if (type.requires === 'series') return hasSeriesField
  if (type.requires === 'second_y') return hasSecondY
  if (type.requires === 'skill_data') return false
  return false
}

## Part 6 — Live Preview

The preview column runs the chart query on every
field change with a 500ms debounce. No "Run query"
button needed.

```typescript
// Debounced query execution:
useEffect(() => {
  const timer = setTimeout(async () => {
    if (!spec.x_field || !spec.y_aggregate) return
    setPreviewLoading(true)
    try {
      const data = await fetchChartData(spec, workspaceId)
      setPreviewData(data)
    } catch (err) {
      setPreviewError('Query failed')
    } finally {
      setPreviewLoading(false)
    }
  }, 500)
  return () => clearTimeout(timer)
}, [spec])
```

Preview loading state: skeleton bars animating in
the chart area.

Title input: above the chart preview. Placeholder:
"State what this chart proves, not what it shows"

Color mode radio group: below the chart preview.
Three options: Semantic / Uniform (default) / Categorical
Changing color mode updates the preview immediately
without re-querying.

function InsertChartTrigger({
  onClick, label
}: { onClick: () => void, label: string }) {
  return (
    <div
      onClick={onClick}
      style={{
        margin: '8px 0 16px',
        padding: '8px',
        border: '1.5px dashed transparent',
        borderRadius: 6,
        textAlign: 'center',
        fontSize: 12,
        color: 'transparent',
        cursor: 'pointer',
        transition: 'all 150ms',
      }}
      onMouseEnter={e => {
        const el = e.currentTarget
        el.style.borderColor = '#14B8A6'
        el.style.color = '#14B8A6'
      }}
      onMouseLeave={e => {
        const el = e.currentTarget
        el.style.borderColor = 'transparent'
        el.style.color = 'transparent'
      }}
    >
      {label}
    </div>
  )
}
```

## Part 8 — handleInsertChart

When the user clicks "Insert chart →", build the
TipTap pandoraChart node and insert it into the
correct section:

```typescript
async function handleInsertChart(
  sectionId: string,
  spec: ChartSpec
) {
  // 1. Find the section's TipTap editor via
  //    sectionEditorRegistry (already built)
  const editor = sectionEditorRegistry.get(sectionId)
  if (!editor) return

  // 2. Insert the pandoraChart node
  editor.chain().focus().insertContent({
    type: 'pandoraChart',
    attrs: { spec: JSON.stringify(spec) }
  }).run()

  // 3. Save the document
  await saveReportSection(sectionId)
}
```

Use the existing sectionEditorRegistry and
insertChartIntoEditor() if they exist from the
earlier Report Viewer build. Match that pattern.
