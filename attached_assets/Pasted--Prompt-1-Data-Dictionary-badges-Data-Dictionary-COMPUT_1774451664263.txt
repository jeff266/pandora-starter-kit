function SourceBadge({ source }: { source: string }) {

  )

## The Four Internal Action Cards

Internal action cards should feel distinct from
CRM action cards. CRM actions write to external
systems (HubSpot, Salesforce). Internal actions
update Pandora's own knowledge — they should
feel lighter and more confirmatory.

### Visual treatment

CRM cards: amber/orange accent (they're external
writes with higher stakes)
Internal cards: teal accent (they're updating
Pandora's understanding — lower stakes, can revert)

### Card specs per action type

```typescript
const INTERNAL_ACTION_CONFIG = {
  update_data_dictionary: {
    icon:        '📖',
    label:       'Update Data Dictionary',
    accentColor: 'teal',
    descriptionFn: (payload) =>
      `Save refined definition for "${payload.term}"`,
    confirmText: 'Update',
    revertable:  true,
  },

  update_workspace_knowledge: {
    icon:        '🧠',
    label:       'Save to workspace knowledge',
    accentColor: 'teal',
    descriptionFn: (payload) =>
      `Remember: ${payload.value?.slice(0, 80)}...`,
    confirmText: 'Save',
    revertable:  true,
  },

  confirm_metric_definition: {
    icon:        '✓',
    label:       'Confirm metric',
    accentColor: 'teal',
    descriptionFn: (payload) => {
      const formatted = payload.unit === 'percent'
        ? `${(payload.value * 100).toFixed(1)}%`
        : payload.unit === 'dollars'
        ? `$${(payload.value/1000).toFixed(0)}K`
        : String(payload.value)
      return `Lock ${payload.metric_key?.replace(/_/g,' ')} at ${formatted}`
    },
    confirmText: 'Confirm',
    revertable:  true,
  },

  update_calibration: {
    icon:        '⚙',
    label:       'Update calibration',
    accentColor: 'teal',
    descriptionFn: (payload) =>
      `Save new threshold for ${payload.dimension_key}`,
    confirmText: 'Update',
    revertable:  true,
  },
}
```

### Card layout

```tsx
function InternalActionCard({
  action,
  onApprove,
  onDismiss
}: {
  action: PendingAction
  onApprove: () => void
  onDismiss: () => void
}) {
  const config = INTERNAL_ACTION_CONFIG[action.action_type]
  if (!config) return null

  return (
    <div className="flex items-start gap-3 p-3
      rounded-lg border border-teal-200
      bg-teal-50/50">

      <span className="text-lg flex-shrink-0 mt-0.5">
        {config.icon}
      </span>

      <div className="flex-1 min-w-0">
        <div className="text-xs font-medium
          text-teal-700 mb-0.5">
          {config.label}
        </div>
        <div className="text-xs text-slate-600
          leading-relaxed">
          {config.descriptionFn(action.payload)}
        </div>
      </div>

      <div className="flex gap-2 flex-shrink-0">
        <button
          onClick={onApprove}
          className="text-xs px-2.5 py-1 rounded
            bg-teal-600 text-white font-medium
            hover:bg-teal-700 transition-colors"
        >
          {config.confirmText}
        </button>
        <button
          onClick={onDismiss}
          className="text-xs px-2.5 py-1 rounded
            border border-slate-200
            text-slate-500 hover:bg-slate-50
            transition-colors"
        >
          Dismiss
        </button>
      </div>
    </div>
  )
}
```

### Integration

In the action card renderer (the "N actions from
this analysis" section), route internal action
types to InternalActionCard and CRM types to
the existing CrmActionCard:

```tsx
function ActionCard({ action, ...props }) {
  const INTERNAL_TYPES = [
    'update_data_dictionary',
    'update_workspace_knowledge',
    'confirm_metric_definition',
    'update_calibration',
  ]

  if (INTERNAL_TYPES.includes(action.action_type)) {
    return <InternalActionCard action={action} {...props} />
  }

  return <CrmActionCard action={action} {...props} />
}
```

### Approve/dismiss wiring

Approving an internal action calls the same
approve endpoint already used for CRM actions:
POST /api/workspaces/:id/actions/:actionId/approve

The backend routes internal types to
approveInternalAction() automatically —
no frontend routing changes needed.

Dismissing calls the existing dismiss/reject
endpoint.
