## Where They Diverge (Decisions Needed)

**1. Assignment mechanism: `assigned_to_user_id` (Replit) vs `owner_email` (mine)**

This is the biggest architectural fork. Replit links targets to Pandora user accounts via a FK to users(id). My spec uses email strings matching the CRM's `owner_email` pattern.

Replit's approach is cleaner long-term (referential integrity, proper joins), but it requires reps to be Pandora users. Today, your reps aren't Pandora users — they're deal owners in HubSpot/Salesforce. Jeff is often the only actual Pandora user per workspace. If you create an Individual target for Nate Phillips and try to set `assigned_to_user_id`, Nate doesn't have a user record to reference.

My recommendation: use Replit's `assigned_to_user_id` as the primary column, but add `assigned_to_email TEXT` alongside it as a CRM-matching fallback. The visibility function checks both — if the user has a Pandora account, match by user_id; if they're a CRM-only rep, match by email against `owner_email` on deals. This keeps the FK clean for users who exist while supporting the reality that most "users" in your system today are CRM email addresses.

**2. Default visibility for no role set: `['company']` (Replit) vs full admin (mine)**

Replit defaults null pandora_role to only company targets. My spec defaults to admin (full visibility). This matters right now because Jeff has no pandora_role set on any workspace, and he's the one asking "what is our annual quota?"

My recommendation: if `pandora_role` is null AND the user is the workspace owner/admin, default to full visibility. Otherwise default to `['company']`. This matches Replit's intent (see their note about overriding) but makes it explicit.
